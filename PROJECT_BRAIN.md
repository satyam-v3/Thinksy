# 🧠 Thinksy: Core System Architecture & Onboarding Brain

**Last Updated:** Production Deployment (Vercel/Render Phase)
**Project Mission:** Transform static PDF documents into an interactive, hallucination-free, AI-powered conversational knowledge base using Retrieval-Augmented Generation (RAG).

Welcome to the Thinksy engineering team. This document contains everything you need to understand our data flows, AI pipelines, and state management so you can start shipping code today.

---

## 1. 🏗️ High-Level Architecture
Thinksy is a decoupled, cloud-native application.

* **Frontend:** React (Vite, TypeScript, TailwindCSS) hosted on **Vercel Edge Network**.
* **Backend:** Node.js (Express, TypeScript) hosted on **Render**.
* **Database & Vector Store:** **MongoDB Atlas** (M0/M10 Cluster with `$vectorSearch` enabled).
* **Blob Storage:** **Cloudinary** (Secure PDF CDN).
* **AI Infrastructure:** **Google Gemini** (LLM) & **Hugging Face** (Embeddings).

---

## 2. 🗄️ Databases & Storage

### MongoDB Atlas (Primary Database)
We use MongoDB for both relational user data and high-dimensional vector storage. This prevents the need to synchronize two separate databases (e.g., Postgres + Pinecone).

* **`User` Collection:** Stores authentication data. Passwords are encrypted via `bcryptjs`.
* **`Chunk` Collection:** Stores the vectorized PDF data. 
    * `userId`: Isolates data per user.
    * `source`: Maps to the Cloudinary `storedFilename` (Crucial for filtering).
    * `text`: The raw string of the PDF chunk.
    * `embedding`: Float array generated by Hugging Face (Indexed for Atlas Vector Search).

### Cloudinary (File Storage)
We do not store PDFs on the Render server disk.
* PDFs are buffered in memory via `multer` and streamed directly to Cloudinary.
* Cloudinary returns a secure URL and a `storedFilename` ID.
* The frontend uses the Cloudinary URL to render the PDF viewer (`react-pdf`).

## 3. 🧠 AI & Retrieval Systems

### 3.1 Embedding Pipeline (Ingestion)
We use **Hugging Face Inference API** to generate embeddings.
1.  **Semantic Chunking:** We do not blindly slice arrays. We split by paragraphs (`\n\n`) and fallback to sentences. We use a **20% sliding window overlap** to ensure entities at the end of a chunk are preserved in the next chunk.
2.  **Vectorization:** Text chunks are batched via `Promise.all()` to prevent API bottlenecks, converted into dense vectors, and upserted to MongoDB.

### 3.2 Retrieval Engine (RAG)
We use **MongoDB Atlas `$vectorSearch`** natively written in C++.
* **Metric:** Cosine Similarity.
* **Filtering:** The aggregation pipeline enforces a strict match on `userId` (security) and `$in: activeDocs` (context scoping). 
* *Note on `activeDocs`:* The frontend passes the Cloudinary `storedFilename` to match the exact `source` field in the database.

### 3.3 Generative LLM
We use **Google Gemini 2.5 Flash**.
* **Prompting:** We inject the retrieved chunks into a strict system prompt. The LLM is instructed to act as a synthesizer, not an oracle. It *must* append citations like `[1]` linked to the provided chunk index.
* **Streaming:** Responses are streamed back to the client token-by-token using **Server-Sent Events (SSE)**.

## 4. 🔄 Core Workflows

### Workflow A: PDF Upload & Vectorization
1. Client selects PDF → POST `/api/v1/pdf/upload` (FormData).
2. `multer` caches the file buffer.
3. Streamed to Cloudinary → Returns `secure_url`.
4. `pdf-parse` extracts raw text.
5. Semantic Chunker splits text into `N` overlapping chunks.
6. Hugging Face converts chunks into `N` embeddings.
7. Mongoose `insertMany` saves chunks to MongoDB.
8. Returns Cloudinary metadata to Frontend UI.

### Workflow B: Chat & Streaming (The RAG Loop)
1. User types query → Frontend fetches JWT from `localStorage`.
2. POST `/api/v1/chat/stream` with `Authorization` header.
3. Backend embeds query via Hugging Face.
4. MongoDB `$vectorSearch` retrieves Top-5 similar chunks.
5. Context + Chat History pushed to Gemini API.
6. Backend opens SSE stream (`text/event-stream`).
7. Tokens stream to frontend → React buffers and renders.

## 5. 🔌 API Endpoints Reference

All routes are prefixed with `/api/v1`. Protected routes require a Bearer JWT.

| Endpoint | Method | Auth Req | Description |
| :--- | :--- | :--- | :--- |
| `/auth/register` | `POST` | No | Creates User, returns JWT. |
| `/auth/login` | `POST` | No | Authenticates User, returns JWT. |
| `/pdf/upload` | `POST` | Yes | Uploads to Cloudinary, extracts text, chunks, embeds, saves to DB. |
| `/chat/query` | `POST` | Yes | Standard REST completion (Used for debugging). |
| `/chat/stream` | `POST` | Yes | Opens SSE connection. Returns tokens and citation metadata. |

## 6. 🛠️ UI & Learning Tools (Frontend)

### Streaming Stability (The rAF Loop)
**⚠️ Critical Knowledge:** LLMs stream up to 50 tokens/sec. Triggering a React state update for a complex Markdown tree 50 times a second will crash the browser. 
* **Solution:** We use a mutable `useRef` buffer to catch incoming tokens. A `requestAnimationFrame` loop flushes the buffer to the React state strictly once per monitor refresh cycle (~60fps). 

### Source Attribution & Citations
* **Regex Parsing:** A custom `react-markdown` plugin intercepts patterns like `[1]`.
* **Interactive `<SourceCard />`:** Clicking a citation scrolls the UI to the exact chunk metadata card.
* **State Mapping:** The `<SourceCard />` checks `localStorage` or context for the matched Cloudinary URL and opens the PDF viewer modal on demand.

## 7. 🗺️ Future Roadmap & Next Steps

When picking up a new ticket, refer to this roadmap:

1.  **Hybrid Search (BM25 + Dense Vectors):** Enhance MongoDB retrieval. Dense vectors are great for semantic meaning ("how loops work"), but BM25 sparse vectors are needed for exact keyword matching (e.g., specific ID numbers or acronyms).
2.  **Asynchronous Queues (Redis/BullMQ):** Move the PDF parsing and embedding pipeline off the main Express thread. Send the user an immediate `202 Accepted` and update the UI via WebSockets when vectorization is complete.
3.  **Active Learning Artifacts:** Create an endpoint that runs a background prompt over an uploaded document to automatically generate flashcards and multiple-choice quizzes, saving them to a new `Quiz` MongoDB collection.
4.  **Multi-Modal OCR (Gemini 1.5 Pro Vision):** Upgrade `pdf-parse` to handle image-heavy PDFs, diagrams, and scanned math equations.
5.  **Maximum Marginal Relevance (MMR):** Update the reranking algorithm to penalize redundant chunks from the same paragraph, giving the LLM a more diverse context window.
